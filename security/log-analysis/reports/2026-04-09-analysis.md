## Executive Summary

During a 25-hour capture window spanning 2026-04-08 to 2026-04-09, 16,417 firewall log entries were recorded across the WAN and IoT VLAN interfaces. Of these, 16,283 (99.2%) were blocked by the default-deny WAN policy. The threat landscape in this window is significantly broader than the prior 5.5-hour capture — with 3,865 unique attacking IPs representing coordinated, multi-vector reconnaissance at scale.

Three new high-priority patterns emerged: a concentrated RDP scanning campaign from two distinct /24 clusters (AS207812 Bulgaria and AS214295 Romania/UK-registered), a systematic sequential port sweep from a single AWS EC2 instance covering 154 unique ports, and probes targeting Docker API (TCP 2375) and Android ADB (TCP 5555) — services that would present critical unauthenticated access risks if exposed. VirusTotal confirmed the top offenders as actively malicious, with reputation scores as low as -9.

A false-positive cluster was identified: all 263 inbound UDP 3478 (STUN/WebRTC) events originate exclusively from Microsoft Azure IP space and represent Microsoft Teams NAT traversal probes — not an attack. A new SOC_SILENCE rule is recommended to suppress this noise class. All external probes were blocked. No lateral movement or internal compromise indicators were observed.

**Analysis:** Consecutive TCP SYN packets (1-second apart, identical sequence number = retransmission) targeting port 5555 — the Android Debug Bridge (ADB) port. Attackers probing for exposed ADB typically seek Android-based IoT devices (smart TVs, set-top boxes) or developer machines with ADB over network enabled. The full TCP options fingerprint (`mss;sackOK;TS;nop;wscale`) indicates a real OS stack, not a stripped scanner. 61 total ADB probes this window. Blocked.

### Reconnaissance: SIP VoIP Enumeration (UDP 5060)

```text
<134>1 2026-04-08T17:53:50-04:00 edgefw.core.home.arpa filterlog - - -
  1,6,,1000000103,ix1,match,block,in,4,0x0,,49,52191,0,DF,17,udp,432,
  94.23.167.143,[REDACTED_WAN_IP],5108,5060,412
```

**Analysis:** UDP probe to port 5060 (SIP — Session Initiation Protocol) with a payload of 412 bytes. SIP enumeration is used to discover PBX/VoIP infrastructure for toll fraud attacks — once a SIP server is found, attackers send REGISTER or INVITE messages with credential brute-force to gain outbound call capability, potentially running up thousands in international call charges. The large payload size (412 bytes) is consistent with a full SIP OPTIONS or REGISTER request, not a basic port probe. Source `94.23.167.143` is OVH GmbH (AS16276, Germany). 53 total SIP probes this window. Blocked.

## SIEM Readiness Notes

- RFC 5424 format confirmed — natively compatible with Splunk, Elastic/ELK, Wazuh, and Microsoft Sentinel.
- ISO 8601 timestamps with UTC-4 offset — no normalisation required for most SIEM platforms.
- Comma-separated `filterlog` payload maps cleanly to field-level parsing; rule ID and tracker fields enable per-rule volume analytics.
- `SOC_SILENCE_*` rule prefix enables SIEM filter rules to exclude noise at ingest without losing audit trail.
- **New enrichment opportunity:** GeoIP tagging would reveal the Netherlands/UK/Eastern Europe cluster concentration visible in this capture — enabling geo-based alert tiering.
- **Correlation rule candidate:** `>10 unique IPs from same /24 targeting port 3389 within 60 minutes` → RDP scanning cluster alert (medium severity). Would have fired on both 45.142.193.x and 79.124.62.x clusters in this window.
- **Correlation rule candidate:** `single source IP → SYN to >50 unique destination ports within 30 minutes` → port sweep alert (high severity). Would have fired on `13.59.59.233`.
- **Automated VT enrichment:** Any source IP exceeding 50 blocked events/hour should trigger automated VirusTotal lookup via API for real-time reputation scoring.

## Recommendations

1. **Deploy `SOC_SILENCE_AZURE_STUN`** — Create a pfSense firewall rule to suppress `52.115.0.0/16` UDP 3478 logging. This eliminates ~10 false-positive log entries/hour with zero security value. Microsoft Teams functionality is unaffected — STUN requests are inherently one-way discovery probes.

2. **Docker API immediate alert** — Create a SIEM or pfSense alert rule that fires on any inbound TCP SYN to port 2375 from any source. Even with default-deny in place, the presence of this probe type signals that external actors are actively hunting for unauthenticated container infrastructure. Severity: high.

3. **RDP cluster detection rule** — Implement SIEM correlation: if ≥5 unique source IPs from the same /24 subnet target port 3389 within a 60-minute window, fire a medium-severity alert. Would have detected both the 45.142.193.x and 79.124.62.x clusters within minutes of campaign onset.

4. **GeoIP enrichment** — Tag all inbound WAN block events with country of origin. Netherlands, Bulgaria, Ukraine, and Canada (Modat B.V.) collectively account for the majority of scanning volume in this window. GeoIP tiers would allow faster triage and the option for region-based elevated alerting.

5. **GRE monitoring** — 120 GRE probes from 60+ unrelated IPs is a persistent and unusual pattern. Add a specific alert for any inbound GRE (IP protocol 47) event. This traffic has no legitimate use case against this network. Recurrence at increasing volume may indicate coordinated VPN/PPTP infrastructure scanning.

6. **Submit to AbuseIPDB** — File community reports for the highest-confidence malicious IPs: `79.124.62.230` (rep -9), `204.76.203.206` (15 VT malicious), and the `85.217.149.0/24` cluster (3,285 hits, 13 VT malicious). This contributes to community threat intelligence and may accelerate blocklisting by downstream consumers.

7. **Extended log capture** — Include DNS query logs from Unbound in the next analysis window. Correlating DNS behaviour with WAN block events would reveal whether any internal device attempted to resolve attacker-controlled infrastructure, which would be a critical lateral movement indicator.
