## Executive Summary

During a 25-hour capture window spanning 2026-04-08 to 2026-04-09, 16,417 firewall log entries were recorded across the WAN and IoT VLAN interfaces. Of these, 16,283 (99.2%) were blocked by the default-deny WAN policy. The threat landscape in this window is significantly broader than the prior 5.5-hour baseline, with 3,865 unique attacking IPs representing coordinated, multi-vector reconnaissance at scale.

Three new high-priority patterns emerged: a concentrated RDP scanning campaign from two distinct /24 clusters (AS207812 Bulgaria and AS214295 Romania/UK-registered), a systematic sequential port sweep from a single AWS EC2 instance covering 154 unique ports, and probes targeting Docker API (TCP 2375) and Android ADB (TCP 5555). Both of those services would present critical unauthenticated access risks if exposed. VirusTotal confirmed the top offenders as actively malicious, with reputation scores as low as -9.

One notable false-positive cluster was identified: all 263 inbound UDP 3478 (STUN/WebRTC) events originate exclusively from Microsoft Azure IP space and represent Microsoft Teams NAT traversal probes, not an attack. A new SOC_SILENCE rule is recommended to suppress this noise class. All external probes were blocked. No lateral movement or internal compromise indicators were observed.

### Risk Assessment

Scanning volume in this 25-hour window equates to ~615 blocks per hour, a 19% increase over the initial 5.5-hour baseline (~518/hr), consistent with sustained internet-wide background noise rather than a targeted campaign. The security posture remains Hardened. All 16,283 external probes were blocked by the default-deny WAN policy with no lateral movement or internal compromise indicators observed. The most significant validated finding was active Docker API (TCP 2375) hunting from multiple global sources, confirming that internet-wide container infrastructure scanning is persistent and that the zero-trust perimeter is actively earning its place.

**Analysis:** UDP probe to port 5060 (SIP) carrying a 412-byte payload. SIP enumeration is used to locate PBX/VoIP infrastructure for toll fraud. Once a SIP server is found, attackers send REGISTER or INVITE messages with credential brute-force to gain outbound call capability, which can rack up significant charges on international calls. A 412-byte payload is consistent with a full SIP OPTIONS or REGISTER request rather than a basic port probe. Source is OVH GmbH (AS16276, Germany). 53 total SIP probes this window. Blocked.

## SIEM Readiness Notes

- RFC 5424 format confirmed, natively compatible with Splunk, Elastic/ELK, Wazuh, and Microsoft Sentinel.
- ISO 8601 timestamps with UTC-4 offset present, no normalisation required for most SIEM platforms.
- Comma-separated `filterlog` payload maps cleanly to field-level parsing rules. Rule ID and tracker fields enable per-rule volume analytics.
- `SOC_SILENCE_*` rule prefix enables SIEM filter rules to exclude noise at ingest without losing the audit trail.
- GeoIP tagging would immediately surface the Netherlands/UK/Eastern Europe cluster concentration visible in this capture and enable geo-based alert tiering.
- Recommended correlation rule: more than 10 unique IPs from the same /24 targeting port 3389 within 60 minutes should fire a medium-severity RDP scanning cluster alert. Both the `45.142.193.0/24` and `79.124.62.0/24` clusters would have triggered this within minutes of campaign onset.
- Recommended correlation rule: a single source IP generating SYN packets to more than 50 unique destination ports within 30 minutes should fire a high-severity port sweep alert. `13.59.59.xxx` would have triggered this early in the capture window.
- Any source IP exceeding 50 blocked events per hour is a good candidate for automated VirusTotal enrichment via API for real-time reputation scoring.

## Recommendations

1. **Deploy `SOC_SILENCE_AZURE_STUN`**: Add a pfSense rule to suppress logging for `52.115.0.0/16` UDP 3478. This cuts ~10 false-positive log entries per hour with no security downside. Teams functionality is unaffected since STUN requests are one-way discovery probes.

2. **Docker API alert**: Add a SIEM or pfSense alert that fires on any inbound TCP SYN to port 2375 from any source. Even with default-deny blocking it, seeing this probe in logs means external actors are actively hunting for unauthenticated container infrastructure on this IP. Severity: high.

3. **RDP cluster detection rule**: Set up SIEM correlation so that 5 or more unique source IPs from the same /24 targeting port 3389 within a 60-minute window fires a medium alert. Both RDP clusters in this capture would have been caught early.

4. **GeoIP enrichment**: Tag inbound WAN block events with country of origin. Netherlands, Bulgaria, Ukraine, and Canada collectively drove most of the scanning volume here. GeoIP would make triage faster and open up region-based alert tiering.

5. **GRE monitoring**: 120 GRE probes from 60+ unrelated IPs is worth watching. Add an alert for any inbound GRE (protocol 47) event since this network has no legitimate use case for it. If volume increases in future captures, it warrants deeper investigation.

6. **Submit to AbuseIPDB**: Report the highest-confidence malicious netblocks: `79.124.62.0/24` (rep -9), `204.76.203.0/24` (15 VT malicious), and `85.217.149.0/24` (3,285 hits, 13 VT malicious). Contributes to community threat intel and may speed up downstream blocklisting.

7. **Extended log capture**: Pull DNS query logs from Unbound into the next analysis window. Correlating DNS lookups with WAN block events would show whether any internal device tried to reach attacker-controlled infrastructure, which would be a strong lateral movement signal.
